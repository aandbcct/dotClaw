# Tool v1 开发计划

## 1. 计划前提

本计划对应《[Tool v1 总体设计](总体设计.md)》。任务类型是“既有 Tool/MCP 模块重构迁移 + 安全能力新增”。用户已确认采用一次性规范命名与配置迁移兼容；不保留旧注册 API 的长期双轨路径。

**实现假设**：在同一开发窗口完成完整目标，但按下列可验证阶段提交。任何阶段不得绕过前一阶段的契约测试。运行代码中的新增注释必须使用中文。

## 2. 现有迁移卡

| 旧项 | 唯一替代者 | 保留期限 | 删除条件 |
| --- | --- | --- | --- |
| `BuiltinToolHandler` | `FunctionToolHandler` + `@tool` 声明 | 仅迁移期间 | 所有 builtin 已由 Discovery 发现，调用方归零，契约测试替代。 |
| `builtin.register_all()` | `ToolDiscovery.discover_builtin()` | 仅迁移期间 | Factory 已改用 Discovery，搜索无生产调用。 |
| 各 builtin `get_*_handler()` | 模块内 `@tool` 函数 | 仅迁移期间 | 所有目标名称注册成功，搜索无生产调用。 |
| `ApprovalManager` 的工具名判定 | `PolicyEngine` + `ApprovalPort` | 可重用交互适配部分可保留 | 无调用依赖旧名称列表，审批适配只接收 `ask` 决策。 |
| MCP resource/prompt handler | MCP 原生 API（不入 Tool Registry） | 本次直接移除注册路径 | `/tools` 与 Context 不再暴露，MCP 原生访问入口不受影响。 |
| MCP 原始 tool 名注册 | `mcp.<server>.<tool>` Adapter | 本次直接替换 | 冲突/命名/调用集成测试通过。 |

## 3. 阶段一：建立基础契约与可验证基线

### 目标

先建立不依赖具体工具业务的类型与契约，避免在迁移 builtin/MCP 时复制 Schema、错误和策略逻辑。

### 新增/修改

- 在 `pyproject.toml` 增加 Pydantic v2 运行依赖。
- 扩展 `tools.base`：明确 ToolDefinition、ToolResult、标准错误码和 ToolExecutionContext 的稳定字段；兼容期保留现有 `output` 读取方式。
- 新增 `tools.decorator`：定义 `@tool` 元数据对象与 `ToolPolicy` 枚举。
- 新增 `tools.schema`：实现 Pydantic JSON Schema 输出、本地原始参数校验、严格未知字段拒绝、校验错误的安全格式化。
- 新增 `tools.function_handler`：只调用已验证本地函数并转换返回值，不再用 `inspect.signature` 在每次调用时猜测 `_context`。
- 改造 `tools.registry`：拒绝重复名称，提供稳定的不可变 definitions 快照；保留显式 `unregister` 以供配置禁用处理。

### 测试与验收

- Pydantic 模型的 JSON Schema、默认值、嵌套模型、枚举和描述转换测试。
- 缺失必填字段、错误类型、未知字段均返回 `INVALID_ARGUMENTS/validation`，且函数未被调用。
- Registry 重名注册测试必须失败并包含双方来源；快照不随后续 Registry 修改而变化。
- Function handler 成功、业务异常、结构化 `ToolResult` 返回的契约测试。

### 风险与完成门槛

- 不要在此阶段改变 builtin 的运行注册路径。
- 仅当新契约单元测试通过且旧测试无无关回归时，进入 Discovery 迁移。

## 4. 阶段二：本地工具装饰器发现与一次性迁移

### 目标

移除手工 `get_*_handler()` / `register_all()`，让 builtin 工具通过可信包自动发现注册。

### 新增/修改

- 新增 `tools.discovery`，只导入并扫描 `dotclaw.tools.builtin` 及其子模块；显式记录导入失败、发现结果与冲突。
- 将 `exec_tool.py`、`file_tool.py`、`memory_tool.py`、`system_tool.py` 改为 `@tool` 声明与 Pydantic 参数模型。
- 将工具名迁移为：
  - `builtin.files.read_text`
  - `builtin.files.write_text`
  - `builtin.files.list_directory`
  - `builtin.process.execute`
  - `builtin.memory.read`
  - `builtin.memory.write`
  - `builtin.system.get_info`
  - `builtin.system.get_time`
- 改造 `agent.factory._build_tools()` 使用 Discovery；Discovery 完成后才应用 `disabled_tools`。
- 删除 `tools.builtin.register_all()`、所有 `get_*_handler()` 和已无调用的 `BuiltinToolHandler`。

### 兼容与配置迁移

- 在配置解析层维护一次性的旧名→新名映射表；对 `disabled_tools`、旧审批列表及同类工具名配置执行转换并输出弃用警告。
- 配置写回（如有）必须使用新名；禁止新增代码读取旧名。
- 明确映射冲突的处理：同一旧名/新名同时出现时，以新名设置为准并记录警告。

### 测试与验收

- 添加一个测试专用 `@tool` 模块，证明新增工具无需改注册列表即可发现。
- 逐个验证八个 builtin 的名称、Schema 与现有业务行为。
- 验证旧配置能转换为新规范名并产生警告；验证新名配置优先。
- 搜索验证：生产代码不再包含 `register_all(`、`get_*_handler`、`BuiltinToolHandler` 的调用或定义（测试中的迁移断言除外）。

### 删除条件

- 所有 builtin 由 Discovery 注册且完整回归测试通过。
- 配置迁移测试覆盖所有旧工具名。
- `rg` 搜索确认旧 API 已无生产引用。

## 5. 阶段三：Capability Broker、Policy Engine 与审批边界

### 目标

建立统一的“验证后才策略判断”的安全链路，替换按工具名审批的逻辑。

### 新增/修改

- 新增 `tools.capability`：定义 `CapabilityRequest`、`CapabilityBroker`、资源类型与 Policy Profile 到请求的映射。
- 新增 `tools.policy`：定义 `PolicyDecision(allow|ask|deny)`、规则合并、全局上限与 Agent 收窄、敏感字段脱敏。
- 对文件 Profile 实现路径规范化与 workspace 根目录检查；显式测试 `..`、绝对路径、符号链接/Windows 联接点逃逸。
- 对进程 Profile 形成受控命令请求，接入危险规则与 `ask`；不在此阶段宣称 OS 级沙箱。
- 对网络和 MCP Profile 形成 host/server 请求，支持 allowlist。
- 将现有 `ApprovalManager` 重构为 Approval Port 的交互适配器：只消费 Policy 的 `ask`，无 Channel 时拒绝。
- 改造 `ToolExecutor` 固定执行顺序：验证 → Broker → Policy → 审批 → handler → Journal。
- 扩展 Journal 工具事件，写入脱敏的策略与审批摘要。

### 配置与默认值

- 扩展 `ToolsConfig` 解析全局 policy 配置和 Agent 收窄配置。
- 实施前由开发者按总体设计给出的 YAML 语义固定默认规则；建议初始值为 `workspace.read=allow`、`workspace.write=ask`、`process.exec=ask`、`network.http=deny`、`mcp.connect=ask`、`mcp.call=ask`。若产品配置与此不同，必须更新设计文档和对应测试。

### 测试与验收

- 校验失败不进入 Broker/Policy/handler。
- workspace 外路径、敏感路径与路径逃逸返回 `POLICY_DENIED`。
- `ask` 有 Channel 时显示已规范化、脱敏后的资源摘要；无 Channel 时返回 `APPROVAL_DENIED`。
- Agent 策略可以收窄而不能放宽全局允许范围。
- 进程、网络、MCP 资源请求与规则命中均有单位测试。
- Journal 不包含密钥、认证头或原始敏感值。

### 风险与完成门槛

- Windows 符号链接/联接点处理必须在真实 Windows CI 或等价测试环境验证。
- 只有所有拒绝路径确认不产生外部副作用后，才将新 Policy 接入真实 builtin。

## 6. 阶段四：MCP 规范化与 Run 级快照

### 目标

让 MCP 成为清晰的远端 Tool Provider，而不是与本地工具行为不一致的旁路。

### 新增/修改

- 改造 `MCPToolProvider`：只发现和注册 MCP tools；移除 resource/prompt 的 Tool Handler 注册流程。
- 改造 `McpToolAdapter`：注册名是 `mcp.<server>.<tool>`，保存原始 MCP tool 名用于 `tools/call`，将 MCP 结果转换为统一 `ToolResult`。
- 将 MCP 参数 Schema 接入 Validator。实现时选用与 Pydantic JSON Schema 一致的校验适配层；不能校验的 schema 必须有明确、保守的错误/降级行为。
- Provider 在 Agent 启动阶段完成首次发现；记录 server 成功/失败状态。重连只更新下一 Run 可见的快照。
- 修改 Runtime/Context 交界，使 Run 创建时获取 immutable Tool Definition 快照，Run 内不读取动态 Registry。
- 将 MCP server 策略授权纳入 Broker/Policy，按 server 单独判断 `mcp.connect` 与 `mcp.call`。

### 测试与验收

- 两个 server 中相同原始 tool 名可同时注册且不冲突。
- MCP resources/prompts 不出现在 `/tools` 或 LLM Tool Definition 中。
- server 连接失败不阻塞 Agent，状态可观测；断线调用返回 `MCP_UNAVAILABLE`。
- Run A 创建后 provider 变化不影响其 definitions；Run B 才看到新快照。
- MCP 参数不合法时不发送 `tools/call`。
- MCP 成功、超时、协议错误、不可用与本地工具具有一致 `ToolResult`/Journal 语义。

### 删除条件

- `McpResourceHandler`、`McpPromptHandler` 及其注册调用均已删除或转为不进入 Tool Registry 的原生 API。
- 搜索验证不再发现未经命名空间的 MCP tool 注册。

## 7. 阶段五：集成、删除与文档收敛

### 目标

完成真实装配、回归验证和旧路径物理删除，确保不存在双轨实现。

### 新增/修改

- 更新 `config.yaml` 示例、`docs/arch/tools-architecture.md`、`docs/wiki/Tool 模块总体说明.md` 与 CLI `/tools` 输出，使其使用新命名、新策略与 MCP 边界。
- 审查 Runtime adapter、Agent factory、Context Tools Slot、CLI `/tools` 与 `/mcp` 的名称/快照依赖。
- 删除已弃用配置字段的读取逻辑，仅保留一次性配置迁移与警告；若项目没有配置持久化，迁移仅发生在加载时。

### 最终删除清单与搜索验证

执行前分别使用 `rg` 验证：

- `register_all`、`get_.*_handler`、`BuiltinToolHandler` 无生产引用。
- `McpResourceHandler`、`McpPromptHandler` 无 Tool Registry 注册引用。
- 旧八个工具名只存在于迁移映射、弃用测试或历史说明中。
- Registry 不再存在“同名静默覆盖”的行为与文案。
- Approval 不再按工具名列表自行决定放行。

### 最终验收

1. 全量默认 pytest 集合通过；新增 tests 纳入默认收集路径或显式调整 pytest 配置。
2. Discovery、Schema、Policy、Approval、MCP、Run 快照、配置迁移和 Journal 均有单元或集成测试。
3. 至少一次手工 CLI 验证：`/tools` 显示新规范名和来源；策略拒绝/审批信息不泄露敏感参数；`/mcp` 正确显示 server 降级状态。
4. 文档、配置示例、错误码与实际代码一致。

## 8. 推荐提交顺序

1. `build(tools): add schema, decorator and registry contracts`
2. `refactor(tools): discover and migrate builtin tools`
3. `feat(tools): add capability broker and policy engine`
4. `refactor(mcp): namespace tools and add run snapshots`
5. `chore(tools): remove legacy registration and update docs`

每个提交都应可运行测试；不要在同一提交中混合无关格式化或业务修改。
