# Agent 数据统计模块 — 需求文档

> 版本：v1.0 | 创建日期：2026-06-07
>
> 研发参考：[数据统计模块设计文档](data-statistics-design.md)

---

## 一、项目背景与动机

### 1.1 问题陈述

dotClaw 是一个基于 ReAct 架构的 AI Agent 框架，其运行效果依赖于多个子系统的协同：
- **ReAct 循环**（推理→行动→观察）
- **工具系统**（工具选择与调用）
- **Skill 系统**（技能触发与脚本执行）
- **记忆系统**（记忆检索与写入）

当对任一子系统进行优化（如调整 prompt、修改工具返回格式、更换记忆策略）后，目前**缺乏客观手段评估优化效果**——只能凭感觉判断"好像快了一点"或"似乎准了一些"。优化决策缺少数据支撑，方向可能跑偏。

### 1.2 核心诉求

| 诉求 | 说明 |
|------|------|
| **量化对比** | 优化前跑一次、优化后跑一次，自动生成对比报告，回答"改完之后到底有没有变好" |
| **瓶颈定位** | 当某个指标异常（如耗时变长），能快速定位是 LLM 推理慢、还是工具调用慢、还是记忆检索慢 |
| **零侵入采集** | 不在业务代码里写埋点，不因为统计模块的加入而影响主线逻辑的可读性和性能 |
| **低成本维护** | 指标体系小而精，砍掉需要人工标注、定义模糊、或纯衍生值的字段，降低采集和存储负担 |

## 二、核心使用场景

### 场景 1：优化前后 A/B 对比

> **作为** dotClaw 开发者
> **我希望** 在修改 prompt / 工具配置 / 记忆策略后，跑同一个测试数据集
> **以便** 获得一份优化前后的指标对比报告，一目了然地看到哪些指标改善、哪些退化

**前置条件**：存在一个固定的测试数据集（如 50 条用户消息），且优化前后配置可复现。

**流程**：
1. 优化前：运行测试数据集 → 产出 `baseline_snapshot.json`
2. 进行代码/配置修改
3. 优化后：运行同一测试数据集 → 产出 `candidate_snapshot.json`
4. 加载两个快照 → 生成对比报告
5. 根据报告判断优化是否生效

### 场景 2：运行时瓶颈诊断

> **作为** dotClaw 开发者
> **我希望** 查看一次运行中各子系统的耗时分布
> **以便** 定位哪个环节是性能瓶颈

**示例**：某次运行端到端延迟偏高，查看快照发现：
- `avg_llm_duration_ms = 1800ms`（占 77%）
- `avg_tool_duration_ms = 540ms`（占 23%）
- `avg_retrieval_ms = 85ms`（可忽略）

→ 结论：瓶颈在 LLM 推理，应优先优化 prompt 长度或切换更快的模型。

### 场景 3：多轮迭代追踪

> **作为** dotClaw 开发者
> **我希望** 保存每次优化后的指标快照
> **以便** 在多次迭代后回溯查看指标的演进趋势

**示例**：
```
v1 快照 → token 用量 3850/任务，成功率 0.89
v2 快照 → token 用量 3135/任务，成功率 0.94  ← 改善
v3 快照 → token 用量 3100/任务，成功率 0.93  ← 持平，略有退化
```

通过快照序列判断优化方向是否正确，何时进入收益递减区。

## 三、功能需求

### FR1：事件采集

| 编号 | 需求 | 优先级 |
|------|------|--------|
| FR1.1 | 系统在 Agent 运行时自动产生结构化事件（如循环开始/结束、工具调用、LLM 请求、Skill 触发、记忆检索），事件格式统一 | P0 |
| FR1.2 | 业务代码只需调用 `emit_event()` 即可发布事件，无需在方法体内编写统计相关逻辑 | P0 |
| FR1.3 | 事件包含时间戳、事件类型、关键数据字段，足够支撑后续所有指标的聚合计算 | P0 |
| FR1.4 | 采集器不影响主流程——事件丢失不应导致 Agent 运行中断 | P1 |

### FR2：指标计算

| 编号 | 需求 | 优先级 |
|------|------|--------|
| FR2.1 | 从一次运行的事件流中，计算并输出包含五大类指标（ReAct 循环、工具、Skill、记忆、通用）的运行快照 | P0 |
| FR2.2 | 快照必须包含以下核心指标：任务完成率、平均循环次数、端到端延迟、token 用量、成本 | P0 |
| FR2.3 | 快照应包含以下诊断指标：工具调用成功率、Skill 触发率、记忆命中率、空转率、P95 耗时 | P1 |
| FR2.4 | 快照不包含需要人工标注才能计算的指标（如工具选择准确率、Skill 触发精准率） | ——（约束） |
| FR2.5 | 快照不包含可从其他字段推导的冗余字段（如占比可从计数计算，总量可从分量求和） | ——（约束） |

### FR3：对比分析

| 编号 | 需求 | 优先级 |
|------|------|--------|
| FR3.1 | 支持加载两份快照（baseline + candidate），自动计算每个指标的变化百分比 | P0 |
| FR3.2 | 自动判断每个指标变化属于"改善"还是"退化"（成功率升高=改善，延迟升高=退化） | P0 |
| FR3.3 | 输出可读的对比报告，按改善/退化分组展示 | P1 |
| FR3.4 | 对比时忽略变化幅度过小（<2%）的指标，避免噪声干扰 | P1 |

### FR4：快照管理

| 编号 | 需求 | 优先级 |
|------|------|--------|
| FR4.1 | 快照以 JSON 格式存储，人类可读，方便手动查看和 diff | P1 |
| FR4.2 | 快照包含运行元信息：运行 ID、时间戳、代码版本（git commit）、配置指纹、测试数据集标识 | P1 |
| FR4.3 | 快照命名规范，支持通过运行 ID 查找和加载 | P2 |

## 四、数据需求

### 4.1 必须采集的运行时事件

| 事件类型 | 触发时机 | 必须携带的数据 |
|---------|---------|-------------|
| `react.loop_start` | ReAct 每轮开始 | 轮次编号 |
| `react.loop_end` | ReAct 每轮结束 | 轮次编号、执行的动作、本轮耗时 |
| `react.empty_action` | LLM 输出了思考但没有行动 | 轮次编号 |
| `tool.call_start` | 工具调用前 | 工具名称 |
| `tool.call_end` | 工具调用后 | 工具名称、是否成功、耗时、错误信息 |
| `skill.trigger` | Skill 被触发时 | Skill 名称、触发来源 |
| `skill.body_loaded` | Skill 内容加载完成 | Skill 名称、注入 token 数、耗时 |
| `skill.script_exec` | Skill 脚本执行 | Skill 名称、脚本路径、是否成功 |
| `memory.retrieval` | 记忆检索 | 是否命中、检索耗时 |
| `memory.write` | 记忆写入 | 记忆类型、是否成功 |
| `llm.request_start` | LLM 请求开始 | 模型名称、输入 token 数 |
| `llm.request_end` | LLM 请求结束 | 模型名称、输出 token 数、耗时 |
| `session.start` | 会话开始 | 会话 ID |
| `session.end` | 会话结束 | 会话 ID、总轮次、是否成功 |

### 4.2 输出指标范围

见 [数据统计模块设计文档](data-statistics-design.md) 第三至七章中精简后的五个 dataclass 定义。

### 4.3 不做什么

- ❌ 不提供实时监控面板 —— 本模块聚焦批量评测对比，不是线上监控系统
- ❌ 不自动判断"质量好坏" —— 不引入 LLM-as-Judge 评估任务结果质量
- ❌ 不采集用户反馈 —— dotClaw 是 CLI 框架，无用户评价机制
- ❌ 不做跨运行长期趋势分析 —— 快照是点状的，趋势分析是快照序列的事，不在本模块范围

## 五、非功能需求

| 编号 | 需求 | 优先级 |
|------|------|--------|
| NFR1 | **零业务侵入**：事件采集通过 Hook/回调机制注入，不修改 Agent 核心循环、工具执行器、记忆系统等模块的业务逻辑 | P0 |
| NFR2 | **采集开销可忽略**：事件对象创建和追加到列表的耗时 < 1ms/事件，不影响 Agent 端到端延迟 | P1 |
| NFR3 | **内存可控**：单次运行（50 任务，每任务平均 3 轮）产生约 500-800 个事件，内存占用 < 5MB | P1 |
| NFR4 | **容错**：事件丢失或采集器异常不应导致 Agent 崩溃或任务失败 | P1 |
| NFR5 | **代码可测试**：采集器、快照构建器、对比逻辑均为纯函数或可 mock 的类，支持单元测试 | P0 |
| NFR6 | **无外部依赖**：不引入数据库、消息队列、监控平台等外部依赖 | P0 |

## 六、验收标准

### AC1：事件采集（FR1）

- [ ] Agent 运行一次完整的测试数据集后，采集器能产出包含全部 14 种事件类型的完整事件列表
- [ ] 移除采集器后，Agent 核心循环代码无任何条件判断或分支逻辑是为了统计而存在的
- [ ] 人为注入事件丢失（如采集器关闭），Agent 正常运行完成

### AC2：指标计算（FR2）

- [ ] 从事件流构建的快照中，五大类指标全部可计算，无 `None` 值（除 `context_overflow_count` 可为 0）
- [ ] 快照字段数与设计文档中精简后的 dataclass 定义严格一致，无遗漏无多余
- [ ] 同一份事件流多次计算产生相同快照（幂等性）

### AC3：对比分析（FR3）

- [ ] 加载两份快照后，对比报告包含所有共有字段的变化百分比
- [ ] 明显改善的指标（如成功率从 0.85 → 0.92）出现在"改善"列表中
- [ ] 明显退化的指标（如延迟从 5000ms → 8000ms）出现在"退化"列表中
- [ ] 变化幅度 < 2% 的指标不出现在改善/退化列表中

### AC4：快照管理（FR4）

- [ ] 快照 JSON 文件可直接用文本编辑器打开阅读
- [ ] 快照包含 `run_id`、`timestamp`、`git_commit`、`config_hash`、`test_dataset` 元信息

### AC5：集成验证

- [ ] 使用优化前的配置跑一次测试数据集，保存快照 A
- [ ] 修改配置（如降低 max_loops、更换模型）后重新跑，保存快照 B
- [ ] 对比 A 和 B，报告能准确反映配置修改带来的指标变化

## 八、术语表

| 术语 | 定义 |
|------|------|
| **快照 (Snapshot)** | 一次完整评测运行产出的所有指标集合，含元信息 |
| **事件 (Event)** | Agent 运行时产生的最小粒度的观测数据，如一次工具调用的开始/结束 |
| **对比报告 (Comparison Report)** | 两份快照的 diff 结果，标注改善/退化 |
| **空转 (Empty Action)** | LLM 输出了思考步骤但没有执行任何工具调用的一轮 ReAct 循环 |
| **冗余行动 (Redundant Action)** | 连续两次调用同一工具且参数完全相同的循环 |
| **重试率 (Retry Rate)** | 同一工具被同一轮循环连续调用超过 1 次的比例 |
| **P95 耗时** | 95% 的样本在该时间内完成，排除 5% 的极端长尾 |
| **TTFT** | Time To First Token，从发出 LLM 请求到收到第一个 token 的耗时 |
