# Runtime 持久化语义与 Session 上下文压缩开发计划

> 状态：已实施并完成验收
>
> 范围：Runtime v2 的 `run.json`、`messages.json`、`events.jsonl`、`session.json`，以及创建 Run 前的 Session 历史压缩。
> 非范围：本阶段不实现每次 LLM 调用前的运行中上下文压缩；仅为其预留可复用契约与迁移路径。

## 1. 背景与问题

当前 `messages.json` 同时保存 Run 内新产生的事实和每次实际发送给 LLM 的完整上下文。
因此同一用户输入会同时以 `user_input` 和 `llm_request(role=user)` 出现，system prompt 与
历史 Conversation 也会在每轮 ReAct 中重复写入。该格式存在以下问题：

- 无法直接看出一次 AgentRun 的上下文如何随工具结果和模型回复演化；
- `RunMessage` 混合了业务事实与调用 payload 副本，语义不唯一；
- 审批恢复需要扫描重复的 `llm_request` 反推历史上下文，恢复逻辑脆弱；
- Session 历史超出预算时由 ContextPort 临时裁剪，Session 本身没有可审计的压缩状态；
- 父子 Agent 的 Task 通信与 Run 执行记录缺少稳定的审计关联。

本阶段的目标是先确定并实现唯一持久化真相：Session 保存长期用户语义历史及其压缩视图，
Run 保存一次执行的冻结初始条件和增量事实，Event 保存外部调用及控制流的审计索引。

## 2. 已确认的架构决策

1. `messages.json` 顶层新增 `initial_context`；其中保存本 Run 开始时冻结的 system context
   与实际注入的历史上下文。
2. `messages` 数组只保存 Run 开始后新增的事实，不再复制 system prompt、历史 Conversation
   或当前用户输入的 LLM payload 副本。
3. `LLM_REQUEST` 不再作为 `RunMessage` 持久化。每次模型调用使用 `events.jsonl` 的
   `LLM_STARTED` 事件审计。
4. system context 按 Slot 结构化保存，不保存无法追溯来源的单一混合文本；当前版本在
   Run 开始时冻结为版本 1。
5. Session 保存原始 Conversation 和版本化压缩摘要。压缩摘要不删除原始 Conversation，
   仅替代后续模型注入的早期历史视图。
6. 本阶段仅在创建 Run 前、取得 Session 租约后判断并执行 `SESSION_HISTORY` 压缩。
   压缩失败时拒绝创建 Run，禁止静默丢弃历史。
7. `ContextCompactionPort` 从首版即支持 `SESSION_HISTORY` 和 `RUN_CONTEXT` 两个作用域；
   本阶段只启用前者，后者留给后续每次 LLM 调用前的压缩迁移。
8. Task 队列负责父子 Agent 的控制通信；Run 负责执行事实；关联信息由 delegation 事件保存。
   目标 Agent 输入只属于子 Run，不在父 Run 复制。

## 3. 容器职责与唯一真相

| 容器 | 唯一问题 | 保存内容 | 禁止保存 |
|---|---|---|---|
| `session.json` | 这个 Session 的长期用户语义历史及其当前压缩视图是什么？ | 原始 Conversation、Conversation 版本、压缩版本链、当前压缩边界 | Run 工具结果、审批状态、LLM payload 副本 |
| `run.json` | 这次执行是谁、何时、为何结束？ | Run ID、Session / 父子归属、状态、统计、消息与 checkpoint 引用 | 完整消息、事件副本、完整上下文 |
| `messages.json.initial_context` | Run 开始时模型的固定初始条件是什么？ | system Slot 快照、压缩历史、近期原始历史、版本与 hash | 后续运行中新产生的事实 |
| `messages.json.messages` | Run 启动后发生了哪些内容事实？ | 用户输入、LLM 回复 / tool call、工具结果、委派结果、最终回复、错误 | 历史副本、完整 LLM 请求 payload |
| `events.jsonl` | 运行中按序发生了哪些调用和控制事实？ | 调用序号、时间、模型、版本、消息引用、hash、Task / child Run 关联、状态变化 | 大 payload、恢复快照、Conversation 副本 |
| `checkpoint.json` | 从哪个安全边界继续、下一步做什么？ | 最小控制状态、消息游标、上下文版本引用、pending 控制、预算 | 完整 prompt、完整工具结果、历史副本 |

## 4. 目标数据格式

### 4.1 Session 的 Conversation 与压缩版本

每条 Conversation 必须具有稳定 `conversation_id`。旧 Session 缺失该字段时，迁移程序或
只读兼容层必须补齐稳定 ID，不能再使用数组下标作为压缩边界。

```json
{
  "id": "session-001",
  "conversation_version": 52,
  "conversations": [
    {
      "conversation_id": "conversation-001",
      "user_query": "用户问题",
      "final_answer": "最终回复",
      "agent_run_ids": ["run-001"],
      "created_at": "2026-07-19T10:00:00+08:00"
    }
  ],
  "active_compression_version": 2,
  "history_compressions": [
    {
      "version": 2,
      "previous_version": 1,
      "covered_through_conversation_id": "conversation-045",
      "content": "截至 conversation-045 的历史摘要",
      "content_hash": "sha256:...",
      "source_conversation_hash": "sha256:...",
      "created_at": "2026-07-19T10:00:00+08:00"
    }
  ]
}
```

新的压缩版本必须由“上一版摘要 + 上一边界之后被覆盖的原始 Conversation”生成。原始
`conversations` 永远保留，以支持审计、重新压缩和迁移。

### 4.2 Run 的初始上下文与增量消息

```json
{
  "run_id": "run-100",
  "version": 2,
  "initial_context": {
    "system_context": {
      "version": 1,
      "slot_order": ["identity", "skills", "memory"],
      "slots": [
        {
          "name": "identity",
          "scope": "static",
          "status": "included",
          "content": "身份和行为约束",
          "content_hash": "sha256:..."
        },
        {
          "name": "memory",
          "scope": "conditional",
          "status": "empty",
          "content_hash": "sha256:..."
        }
      ],
      "rendered_content_hash": "sha256:..."
    },
    "history": {
      "source_session_id": "session-001",
      "source_conversation_version": 52,
      "compressed_history": {
        "compression_version": 2,
        "covered_through_conversation_id": "conversation-045",
        "content": "历史摘要",
        "content_hash": "sha256:..."
      },
      "recent_messages": [
        {"conversation_id": "conversation-046", "role": "user", "content": "..."},
        {"conversation_id": "conversation-046", "role": "assistant", "content": "..."}
      ],
      "truncation_applied": false,
      "content_hash": "sha256:..."
    }
  },
  "messages": [
    {"id": "input-1", "sequence": 1, "kind": "user_input", "role": "user", "content": "本次输入"},
    {"id": "response-1", "sequence": 2, "kind": "llm_response", "role": "assistant", "tool_calls": []},
    {"id": "tool-1", "sequence": 3, "kind": "tool_result", "role": "tool", "content": "工具结果"},
    {"id": "response-2", "sequence": 4, "kind": "final_response", "role": "assistant", "content": "最终回复"}
  ]
}
```

当前版本 system context 的 Slot 集合和内容在 Run 内冻结。未来若 Slot 内容、顺序或集合变化，
必须追加新的 system context 版本，后续 `LLM_STARTED` 明确引用该版本，不得覆写版本 1。

### 4.3 LLM 调用事件

`LLM_STARTED` 是模型输入审计记录，不是消息。它不复制完整 prompt，而是保存可复原上下文
的版本与引用。

```json
{
  "run_id": "run-100",
  "sequence": 8,
  "event_type": "llm_started",
  "data": {
    "call_index": 2,
    "model_id": "model-x",
    "system_context_version": 1,
    "history_version": 1,
    "incremental_message_ids": ["input-1", "response-1", "tool-1"],
    "context_hash": "sha256:...",
    "tool_schema_hash": "sha256:..."
  }
}
```

任意 `incremental_message_ids` 都必须存在于同 Run 的 `messages.json.messages`。初始 system
和历史通过版本引用，不伪造为 RunMessage ID。

### 4.4 委派与 Task 队列关联

父 Run 的 assistant `tool_calls` 是委派目标、标题和 objective 的唯一执行事实；
`DELEGATION_SUBMITTED` 事件保存 `task_id`、`child_run_id`、目标 Agent 与目标 Session。
子 Run 将收到的 objective 持久化为自己的 `USER_INPUT`；父 Run 只保存
`DELEGATION_RESULT` 及其关联 ID。

当前 `TaskMessageBroker` 是进程内 Task 通信通道，不应成为第二份持久化消息真相。未来即使
Broker 改为持久化队列，Task 记录也只保存调度通信，Run 文件仍保存实际执行事实。

## 5. 上下文压缩契约

### 5.1 通用 Port

```python
class ContextCompactionScope(StrEnum):
    """压缩摘要所属的上下文范围。"""

    SESSION_HISTORY = "session_history"
    RUN_CONTEXT = "run_context"


class ContextCompactionPort(Protocol):
    """将有序上下文片段压缩为可版本化摘要。"""

    async def compact(
        self,
        request: ContextCompactionRequest,
    ) -> ContextCompactionResult:
        """基于已有摘要和待覆盖片段生成新的压缩结果。"""
```

`ContextCompactionRequest` 至少包含：作用域、上一版摘要、待覆盖的有序片段、目标 token
预算、来源版本和内容 hash。结果至少包含：新版本、覆盖边界、摘要内容、来源 hash 与模型
元数据。所有枚举语义必须以枚举类型表达，禁止使用散落字符串。

### 5.2 本阶段启用：Session 历史压缩

创建 Run 前，在 Session 租约内由 `SessionHistoryPreparationService` 执行：

```text
接收用户输入
→ 获取 Session 租约
→ 计算 system / 工具 schema / 当前输入预留预算
→ 判断历史轮次或历史 token 是否超限
→ 调用 ContextCompactionPort(scope=SESSION_HISTORY)
→ 原子更新 session.json 的压缩版本链
→ 冻结包含压缩摘要和近期消息的 RunRequest
→ 创建并执行 AgentRun
```

触发条件：超过最大历史轮次，或在预留 system、工具定义和当前用户输入后，历史部分仍超过
可用 token 预算。压缩失败、写入失败或压缩后仍无法满足预算时，返回明确错误并拒绝启动 Run；
禁止回退为静默裁剪。

### 5.3 后续迁移：每次 LLM_STARTED 前的运行中压缩

后续版本必须复用同一 `ContextCompactionPort` 和预算规划器，不重新设计接口：

```text
每次 LLM_STARTED 前
→ ContextBudgetPlanner 评估 initial_context + 当前 Run 增量消息
→ 若超预算，调用 ContextCompactionPort(scope=RUN_CONTEXT)
→ 在 Run 内追加新的上下文摘要版本和覆盖范围
→ LLM_STARTED 引用新的 system / history / run context 版本
→ 调用 LLM
```

`RUN_CONTEXT` 压缩只能保存到当前 Run，不能写入 `session.json`。工具结果、中间模型回复和
审批前状态可能属于失败或取消的 Run；它们在成功投影前不是 Session 用户语义历史。

本阶段需要预留但不启用的内容：

- `ContextCompactionScope.RUN_CONTEXT` 枚举值与 DTO；
- Run 内上下文摘要版本的数据模型和 Repository 扩展点；
- Checkpoint 对“最后生效上下文版本”的引用字段；
- `LLM_STARTED` 按版本重建模型输入的测试夹具。

│   │   └── session_run_coordinator.py # Session 租约内编排准备与执行

## 7. 实施阶段

### 阶段 A：格式契约与兼容读取

**新增**：初始上下文、Slot 快照、历史快照、压缩作用域和版本化压缩摘要的数据模型；
`messages.json` 格式版本 2。

**修改**：RunRepository 的消息序列化 / 反序列化、事件引用校验、RunMessageKind 与
RunEventType。

**废弃**：新写入路径中的 `RunMessageKind.LLM_REQUEST`。

**删除条件**：旧格式迁移或只读兼容测试通过；所有新 Run 不再写 `llm_request` message。

### 阶段 B：Session 历史压缩与冻结请求

**新增**：`ContextCompactionPort`、预算规划器、SessionHistoryPreparationService、LLM
压缩 Adapter、压缩版本链写入。

**修改**：Conversation 稳定 ID、Session 原子更新、Agent / Coordinator 在租约内创建
RunRequest 的时序。

**废弃**：ContextPort 内将历史超限直接静默截断的行为。

**删除条件**：首次压缩、滚动压缩、压缩失败、并发 Session 与旧 Session 升级测试通过。

### 阶段 C：Runtime 消息与事件迁移

**新增**：首次模型调用前 `initial_context` 冻结与 `LLM_STARTED` 写入；委派关联事件字段。

**修改**：RuntimeEngine 不再把 ContextBundle 的全部消息写入 `messages.json`；审批恢复按
初始快照和增量消息重建上下文。

**废弃**：通过扫描重复 `llm_request` 重建审批上下文的逻辑。

**删除条件**：普通对话、多工具、审批恢复、取消和 delegation 的上下文重建测试全部通过。

### 阶段 D：迁移、审计与文档收口

**新增**：旧 `messages.json` 的迁移命令或明确只读兼容层；迁移报告。

**修改**：Runtime 设计、迁移清单、持久化说明和测试夹具。

**废弃**：旧格式新写入路径及其无效测试。

**删除条件**：全量回归、格式校验、迁移样例、`git diff --check` 均通过；生产源码没有旧写入
调用方。

#### 阶段 D 落地规则

v1 `messages.json` 保留**只读兼容**，仓储禁止原地调用 `save_messages()` 或
`save_initial_context()` 改写它。这样不会把旧格式中每轮重复的请求内容误写成一个缺少
`initial_context` 的伪 v2 文件。

需要继续恢复、审计或写入旧 Run 时，先执行以下显式迁移命令：

```powershell
.\.venv\Scripts\python.exe scripts/migrate_messages_v1_to_v2.py data/sessions/{session_id}/agent_runs/{run_id}
```

命令会在同目录保留 `messages.json.v1.bak` 和（如存在）`events.jsonl.v1.bak`，再原子写入：

1. 从首个 `CONTEXT_BUILT` 事件的完整请求副本冻结 `initial_context`；
2. 删除所有被 `CONTEXT_BUILT` 引用的重复 ContextBundle 消息，并重排真实增量消息序号；
3. 删除旧 `CONTEXT_BUILT` 事件，并在每个 `LLM_COMPLETED` 前补写可复原的 `LLM_STARTED`；
4. 再次执行只报告 `already_current`，不修改 v2 文件或覆盖备份。

`messages.json` 与 `events.jsonl` 分别原子替换；若第二个文件写入失败，迁移命令会立即从
上述备份补偿两个文件，使运行目录回到可读取的 v1 状态。

若 v1 文件包含重复 `LLM_REQUEST` 但缺少 `CONTEXT_BUILT` 事件，迁移命令会明确拒绝处理，
保持原文件只读并提示人工处理。没有该索引无法区分后续轮次中复制的 assistant / tool 消息，
继续迁移会破坏“增量消息唯一事实”的约束。

## 8. 写入、恢复与失败语义

### 8.1 正常执行

1. 在 Session 租约内完成历史压缩（如需要）并原子保存 Session。
2. 创建 Run 摘要，冻结并原子保存 `initial_context`。
3. 原子追加增量 `messages`，随后追加仅引用已存在消息的事件。
4. 每次调用前写 `LLM_STARTED`，调用完成后保存回复消息并写 `LLM_COMPLETED`。
5. 成功提交继续沿用成功提交意图：RunEvent、Conversation 投影与 `run.json=COMPLETED`
   必须可幂等补偿。

### 8.2 审批恢复

恢复不得读取变化后的 Session 历史，也不得从重复 payload 推断初始上下文。恢复输入必须是：

```text
messages.json.initial_context
+ 已保存的增量 messages
+ checkpoint 的控制状态与上下文版本引用
```

### 8.3 失败与并发

- 压缩模型失败、内容校验失败、摘要写入失败：拒绝启动 Run，Session 保持旧版本不变；
- Run 失败、取消或等待审批：不得将 Run 内工具 / 中间 LLM 内容写入 Session 压缩摘要；
- 同一 Session：历史准备、压缩写入与 RunRequest 冻结必须在同一租约内串行；
- 不同 Session：允许并行，各自使用独立压缩版本和文件写入；
- 任一 Event 引用的消息必须先完成 `messages.json` 原子写入。

## 9. 测试与完成门槛

必须新增或更新以下测试：

1. `messages.json` v2 格式：初始快照与增量消息分离，禁止写入历史 / system 副本；
2. Slot 审计：Slot 顺序、included / empty / failed 状态、hash 与渲染内容一致；
3. `LLM_STARTED`：调用序号、版本、增量消息引用、hash 与实际 ContextBundle 一致；
4. Session 压缩：首次压缩、滚动压缩、覆盖边界、旧 Conversation 保留、原子失败回滚；
5. 压缩失败：不创建 Run、不修改 Session 当前压缩版本、不发生静默裁剪；
6. 审批恢复：仅使用冻结 `initial_context` 与增量消息，Session 后续变化不影响恢复；
7. delegation：Task ID、child Run ID、子 Run 输入与父 Run 结果可双向追溯；
8. 旧格式：迁移或只读兼容明确、可重复执行、错误信息可行动；
9. 架构约束：Domain 无 Application 依赖，RuntimeEngine 无 LLM / 文件 / Session / Broker 具体实现；
10. 全量测试、`git diff --check` 和持久化样例人工审计通过。

## 10. 本阶段完成定义

满足以下全部条件时，本阶段完成：

- 新 Run 的 `messages.json` 能清晰展示初始条件和后续上下文演化，且不存在每轮重复的
  system / 历史 payload；
- `events.jsonl` 能说明每次 LLM 调用及父子 Agent 调度关联，而不复制大内容；
- Session 以可审计、可版本化的方式压缩历史，压缩失败不会损失上下文；
- `run.json`、`messages.json`、`events.jsonl`、`checkpoint.json` 与 `session.json` 各有唯一职责；
- 接口可直接扩展到后续 `RUN_CONTEXT` 压缩，无需推翻本阶段的模型或文件格式。
