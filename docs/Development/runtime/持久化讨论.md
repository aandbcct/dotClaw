## 1. 这几个部分应分别做什么

先把“log”这个词拆开。它太泛了，建议在代码里不要再把它作为独立概念。

```
RunExecution
    │ 发布 RunEvent
    ▼
AgentRunRepository
    ├─ AgentRun（运行摘要）
    ├─ RunEvent（事实明细，唯一事实源）
    └─ Checkpoint（可恢复快照）
    │
    └─ TraceSink
         └─ Trace（由事件渲染出的调试视图）
```

| 部分           | 是否持久化 | 核心职责                                                     | 是否事实源                 |
| -------------- | ---------- | ------------------------------------------------------------ | -------------------------- |
| `AgentRun`     | 是         | 一次运行的索引和最终摘要：ID、agent/session、起止时间、状态、最终结果引用、错误摘要 | 是，但只存摘要             |
| `RunEvent`     | 是         | 运行中发生了什么的追加式结构化事实：状态迁移、LLM 调用、工具调用、审批、失败、取消等 | **是，唯一明细事实源**     |
| `Checkpoint`   | 是         | 恢复执行所需的某一时刻状态快照                               | 否，可被后续快照替代或清理 |
| `Trace`        | 可选       | 给开发者/UI/排障看的可读时间线、格式化日志                   | 否，可由事件重建           |
| `Conversation` | 是         | 给用户和 LLM 看的语义对话：用户输入与已成功提交的最终回复    | 否，不记录执行过程         |

所以，`AgentRun` 不应该自己再内嵌一份“事件序列”的完整副本。它只保存 `run_id` 与事件范围/数量等摘要；事件通过 `run_id` 关联查询。

换成更清晰的数据关系就是：

```
AgentRun(run_id="r-123", status=COMPLETED, final_message_id="m-9")
    ├─ RunEvent(run_id="r-123", seq=1, type=RUN_STARTED)
    ├─ RunEvent(run_id="r-123", seq=2, type=CONTEXT_BUILT)
    ├─ RunEvent(run_id="r-123", seq=3, type=LLM_COMPLETED)
    ├─ RunEvent(run_id="r-123", seq=4, type=TOOL_COMPLETED)
    └─ RunEvent(run_id="r-123", seq=5, type=RUN_COMPLETED)

Checkpoint(run_id="r-123", seq=4, payload=...)
Trace(run_id="r-123") = 上面事件的可读投影
```

## 2. 你现有模块会怎样变化

你当前已有的三套机制并非要推倒，而是要把职责重新归位：

| 现有模块                                                 | 现状问题                                                     | 重构后的去向                                                 |
| -------------------------------------------------------- | ------------------------------------------------------------ | ------------------------------------------------------------ |
| `session/agent_run.py` 的 `AgentRun` / `AgentRunManager` | 当前既保存运行信息，也序列化消息，容易继续膨胀成全部运行数据容器 | 保留为 Run 摘要与仓储入口；增加独立 `RunEventRepository`，消息按“会话消息 / 执行消息”分开处理 |
| `runtime/state_store.py`                                 | 已有恢复快照，但需要明确它不是最终事实记录                   | 保留并改名或定位为 `CheckpointRepository`；按 `run_id + sequence` 保存，支持只保留最新或关键 checkpoint |
| `journal/journal.py`                                     | 当前既采集事件、维护内部可变状态、写 trace、还能被 Runtime 读取私有字段 | 拆角色：Runtime 只发布 `RunEvent`；Journal 逐步退化为 `TraceSink`/事件订阅器；Runtime 不再读 `_events`、`_agentrun_id`、`_agentrun_sequence` |
| `journal/sinks/trace.py`                                 | 已接近正确方向                                               | 保留为 `TraceSink`，输入改为统一 `RunEvent`，只写可读 trace，不承担恢复和业务判断 |
| `journal/sinks/state_sink.py`                            | 观测模块写恢复状态，职责反转                                 | 从 Journal sink 中移出，改由 `RunRepository` / `CheckpointRepository` 在 Runtime 明确提交点调用 |
| `runtime/agent_state.py`                                 | 纯状态机的方向是对的                                         | 保持纯领域对象；其快照由 Checkpoint 保存，其状态变化发布为 `STATE_TRANSITION` 事件 |
| `runtime/runtime.py`                                     | 同时直接调用 Journal、StateStore、AgentRunManager            | 改为只依赖 `RunRepository`、`CheckpointRepository`、`EventSink` 等端口 |

最重要的一刀是：

> **状态机快照不能经由 Journal 的 `state_sink` 保存。**

原因是恢复状态是执行事务的一部分，不是观察副作用。Runtime 在明确安全点调用 `checkpoint_repository.save(...)`；同时发布一个 `CHECKPOINT_SAVED` 事件给 Trace 即可。

你不需要一开始就真的建立事件表。第一阶段可以让 `AgentRunManager` 内部管理 `AgentRun + events` 两个 JSON 文件或两个 SQLite 表，但接口必须先分开：

```
class RunRepository(Protocol):
    async def create(self, run: AgentRun) -> None: ...
    async def append_event(self, event: RunEvent) -> None: ...
    async def finish(self, result: RunResult) -> None: ...

class CheckpointRepository(Protocol):
    async def save(self, checkpoint: Checkpoint) -> None: ...
    async def load_latest(self, run_id: str) -> Checkpoint | None: ...
```

这样以后存储格式变了，Runtime 不需要改。

## 3. 为什么不建议默认保存“模型推理链”

这里要区分三类东西：

| 内容                                       | 建议                             |
| ------------------------------------------ | -------------------------------- |
| 用户可见文本、工具调用、工具结果、状态迁移 | 默认可记录，按脱敏与保留策略处理 |
| 模型返回的简短解释、计划、进度说明         | 可以记录，并可选择展示给用户     |
| 原始隐藏推理链 / reasoning tokens          | 不建议作为默认持久化内容         |

不是说“绝对不能保存任何中间过程”，而是原始推理链不适合作为运行事实或客户功能的默认数据，原因是：

- 很多模型/API根本不会稳定返回完整推理；不同供应商格式不同，不能把它当作可靠契约。
- 原始推理常含有无效尝试、内部提示词片段、敏感上下文或工具返回的隐私数据，保留和展示风险很高。
- 它并不等于可解释性：一段很长的内部推演未必能帮助客户理解“做了什么、为什么失败、下一步怎么处理”。
- 它会显著扩大存储、检索、权限控制和数据删除的成本。

如果客户需要“过程透明”，我建议你把它设计成**可解释执行记录**，而不是暴露原始 CoT：

```
用户目标
  → 执行计划摘要（可选）
  → 工具调用与结果摘要
  → 阶段进度
  → 关键决策理由摘要
  → 最终结果 / 失败原因与可恢复建议
```

这些内容可由 Runtime 的结构化事件提供，例如：

```
{
  "type": "TOOL_COMPLETED",
  "tool": "search_code",
  "outcome": "success",
  "summary": "找到 12 个 Runtime 相关调用点",
  "user_visible": true
}
```

如果你确实希望为调试保存模型的 reasoning 字段，可以做成严格 opt-in 的 `debug.reasoning_capture`：

- 仅保留供应商明确返回、允许保存的字段；
- 默认关闭；
- 不注入 Conversation；
- 独立加密/访问权限/短 TTL；
- Trace 展示默认仍用摘要，而不是原文；
- 对多租户或客户环境按租户策略决定是否允许。

这样既能满足高级调试与审计需求，也不会让“客户想看过程”变成把不稳定、敏感的原始模型思考永久写进核心数据模型。
