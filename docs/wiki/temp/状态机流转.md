## 状态机事件流转 → Runtime 操作节点路由

整个链路分三层：**Agent（入口门面）→ Runtime（执行引擎）→ AgentState（事件驱动状态机）**。核心循环在 `Runtime._step()` 里。

### 一、整体架构

code复制

```
用户消息
  │
  ▼
Agent.process(runtime, session, msg)
  │
  ▼
Runtime.run(session, agent, msg)
  │
  ▼
Runtime._step()  ←── 这是 ReAct 循环的主战场
  │
  ├─ 创建 AgentState(phase=IDLE)
  ├─ feed AgentStartEvent → 状态机返回 AgentAction.INVOKE_LLM
  │
  └─ while not state.is_terminal:
       ├─ INVOKE_LLM   → 调 LLM → feed LLMResponseEvent → 路由
       ├─ EXECUTE_TOOLS → 执行工具 → feed ToolsDoneEvent → 路由
       ├─ WAIT          → 特判 TRUNCATED/WAITING_APPROVAL / RETRYING
       ├─ FINALIZE      → 提取结果，退出循环
       └─ HANDOFF_TARGET→ 创建子 Agent，递归 run()
```

### 二、状态机核心：TRANSITION_TABLE 声明式路由

所有 `(phase, event) → (next_phase, action)` 集中在 `agent_state.py` 的 `_TRANSITION_TABLE` 中声明。`handle_event()` 按 **priority 越小越优先** 匹配规则，执行 side_effect 后再通过 `_auto_chain()` 自动推进 ContinueEvent 链。

关键转换如下：

#### 1. 启动：IDLE → THINKING

```
Event: AgentStartEvent(user_message)
→ THINKING, INVOKE_LLM
```

Runtime 收到 `INVOKE_LLM`，调用 LLM。

#### 2. THINKING 分支（LLMResponseEvent）

| priority | guard                            | → next_phase | → action      |
| :------- | :------------------------------- | :----------- | :------------ |
| 10       | `finish_reason="error"`          | FAILED       | FINALIZE      |
| 20       | `finish_reason="length"`（截断） | TRUNCATED    | **WAIT**      |
| 30       | 有 tool_calls                    | ACTING       | EXECUTE_TOOLS |
| 40       | 默认（纯文本回复）               | RESPONDING   | FINALIZE      |

**RESPONDING 是瞬态**，`_auto_chain()` 自动推 ContinueEvent → DONE → FINALIZE。Runtime 收到 FINALIZE 后退出循环。

**TRUNCATED 返回 WAIT**，Runtime 在 WAIT 分支中注入 “请继续” 提示后推 ContinueEvent：

- v2 续跑：`_guard_truncated_continue` 通过 → THINKING，回到 ReAct 循环
- v1 兼容：直接 → RESPONDING → 结束

#### 3. ACTING 分支（ToolsDoneEvent）

| priority | guard                       | → next_phase | → action       |
| :------- | :-------------------------- | :----------- | :------------- |
| 5        | retryable error             | RETRYING     | WAIT           |
| 10       | tool error                  | FAILED       | FINALIZE       |
| 20       | handoff 信号                | HANDOFF      | HANDOFF_TARGET |
| 30       | stop 信号                   | RESPONDING   | FINALIZE       |
| 40       | 安全阀（超迭代/工具死循环） | RESPONDING   | FINALIZE       |
| 50       | 默认（继续下一轮）          | THINKING     | INVOKE_LLM     |

正常路径是 priority 50：工具执行完无异常 → 回到 THINKING → 下一轮 INVOKE_LLM，实现 ReAct 多轮迭代。

**RETRYING 返回 WAIT**，Runtime 自动推 ContinueEvent：

- 重试未达上限 → THINKING（带错误信息重试）
- 重试耗尽 → RESPONDING → 结束

#### 4. HANDOFF → HANDOFF_TARGET

Runtime 收到 `HANDOFF_TARGET` 后，调用 `_handle_handoff_for_state()` → `derive()` 派生子 Runtime → 递归 `run()` 子 Agent。

### 三、Runtime 消费 AgentAction 的完整映射

code复制

```
AgentAction           Runtime 行为
─────────────────────────────────────────────
INVOKE_LLM        → _invoke_llm() → feed LLMResponseEvent
EXECUTE_TOOLS     → _execute_tools_for_state() → feed ToolsDoneEvent
WAIT              → 特判 phase:
                    - TRUNCATED: 注入continue提示 → ContinueEvent
                    - RETRYING: 直接 → ContinueEvent
                    - WAITING_APPROVAL: 持久化 → return WAIT_SENTINEL
FINALIZE          → 提取 final_answer, break 出循环
HANDOFF_TARGET    → _handle_handoff_for_state() → 子Agent递归
```

### 四、关键设计要点

1. **声明式转换表**：所有 `(phase, event, guard, priority)` 集中一处，新增状态无需动 Runtime 代码
2. **瞬态自动推进**：RESPONDING/HANDOFF/TRUNCATED/RETRYING 等 transient phase 通过 `_auto_chain()` 在一次 `handle_event()` 内自动推进到终止态，避免 Runtime 需要多跳
3. **WAIT 哨兵机制**：WAITING_APPROVAL 场景下返回 `WAIT_SENTINEL`，Runtime 持久化状态后退回，调用方稍后通过 `resume_state + resume_event` 恢复执行
4. **安全阀内建于状态机**：`_guard_safety` 在 ACTING 阶段检测 max_iterations 超限和工具死循环，直接走 RESPONDING 结束
5. **ReAct 循环天然闭环**：ACTING → (default) → THINKING → INVOKE_LLM → ACTING，直到 LLM 返回无工具调用的纯文本（走 RESPONDING → DONE）

整个设计很干净：**状态机只管"根据当前 phase + 事件决定下一步是什么"**，所有实际执行（调 LLM、跑工具、handoff、持久化）全部由 Runtime 消费 AgentAction 来完成。两者职责边界清晰，扩展新状态只需在 TRANSITION_TABLE 加一行。

## 护卫条件和副作用

护卫条件（Guard）和副作用（Side Effect）是状态机转换规则的两个核心组成部分，设计上把"判断逻辑"和"执行逻辑"完全解耦了。

### Guard（护卫条件）—— “能不能走这条路”

就是一个签名为 `(state, event) → bool` 的函数。它**只读不写**，纯粹判断当前状态下收到这个事件是否满足转换条件。

以 `_TRANSITION_TABLE` 里的例子说明：

python复制

```python
# THINKING 收到 LLMResponseEvent 时有 4 条规则，靠 guard 区分
TransitionRule(THINKING, LLMResponseEvent,
    guard=_guard_llm_error,      # "finish_reason == 'error' 吗？"
    ...→ FAILED
),
TransitionRule(THINKING, LLMResponseEvent,
    guard=_guard_truncated,      # "finish_reason == 'length' 吗？"
    ...→ TRUNCATED
),
TransitionRule(THINKING, LLMResponseEvent,
    guard=_guard_has_tool_calls, # "LLM 返回了 tool_calls 吗？"
    ...→ ACTING
),
TransitionRule(THINKING, LLMResponseEvent,
    guard=None,                  # 兜底：以上都不满足
    ...→ RESPONDING
),
```

同一个 `(THINKING, LLMResponseEvent)` 组合对应 4 条规则，靠 guard 区分走哪条。`guard=None` 表示无条件兜底匹配。`priority` 越小越先评估，所以是：

> error（10） > truncated（20） > 有工具调用（30） > 纯文本兜底（40）
